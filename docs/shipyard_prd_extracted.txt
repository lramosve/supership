Before You Start: Pre-Search (4 Hours) 
Before writing any code, complete the Pre-Search methodology at the end of this document. This 
structured process uses AI to research real coding agent architectures, define your own approach, and 
lock architecture decisions before you are under time pressure. Your Pre-Search output becomes part of 
your final submission. 
This week emphasizes careful system design over fast shipping. Pre-Search is where you decide what 
your agent does and how — not a box to check on the way to coding. 
 
Background 
Claude Code exists. Cursor exists. Both of them solve the same hard problem: an AI that can read a 
codebase, understand what needs to change, and make targeted edits without breaking everything 
around them. 
The engineering is less obvious than it looks. Rewriting entire files is easy. Making a targeted change to 
one function in a 2,000-line file — without touching anything else, without losing context, without 
hallucinating imports — requires deliberate design. Keeping an agent running continuously, coordinating 
multiple agents in parallel, knowing what context to inject and when: these are real problems that 
different tools solve differently, and the decisions you make early determine whether your agent is 
useful or a liability. 
Your job is to build your own version. Design and implement a coding agent from scratch, then use it to 
rebuild the Ship app. The rebuild is your integration test — it proves your agent works under real 
conditions. When it's done, produce a rigorous comparative analysis: how does the version your agent 
built compare to the original? Where did it fall short? Where did it surprise you? What would you 
change? 

There is no prescribed architecture. There is a defensible one. Finding it, justifying it, and building it is 
the assignment.

File Editing 
This requirement gets its own section because it is the most commonly skipped or faked. An agent that 
rewrites entire files on every change is not a coding agent — it is a code generator with a file system 
wrapper. It will be evaluated accordingly. 
Strategies to research and choose from: 
•​
Unified diff (git-style patches): generate diffs, apply with standard patch tooling. Precise and 
auditable, but requires the LLM to produce well-formed diffs consistently. 
•​
Line-range replacement: identify start and end line numbers, replace only that range. Simple to 
implement, fragile if line numbers drift. 
•​
AST-based editing: parse the file to an abstract syntax tree, modify the target node, serialize 
back. Maximally precise, but language-specific and complex to implement. 
•​
Anchor-based replacement: use unique string anchors within the file to locate and replace 
specific blocks. More robust than line numbers, less complex than AST. 
 
Pick one strategy, justify your choice in PRESEARCH.md, and implement it correctly. Switching strategies 
mid-week because you did not think it through is a planning failure. 
 
Multi-Agent Coordination 
Your system must support spawning and coordinating at least two agents. Document your orchestration 
model: how agents communicate, how outputs are merged, and how conflicts are resolved. LangGraph is 
recommended; any other framework is permitted if you produce equivalent run traces. 
 
Ship App Rebuild 
Once your agent is functional, use it to rebuild the Ship app — all current features — from scratch. You 
are not building a clone by hand. You are directing your agent to build it, then documenting and 
analyzing what it produces. 

The rebuild serves two purposes. First, it is your integration test: if your agent cannot complete a real 
build task, your agent is not done. Second, it generates the data for your comparative analysis. Every 
shortcoming your agent has will surface here — that is the point. 
You are expected to intervene when the agent gets stuck, but document every intervention. 
Interventions are data, not failures. 
 
Comparative Analysis

the original. All seven sections are required. This is the most heavily weighted deliverable — honest, 
specific analysis of a flawed agent scores higher than vague praise of a polished one.

Observability 
Every agent run must be traceable. You must be able to answer after any run: what did the agent do, in 
what order, with what inputs, and what did it produce at each step? LangSmith tracing is recommended 
if you are using LangGraph — it traces automatically once environment variables are set. If you are using 
a different framework, you are responsible for producing equivalent traces.

Build Strategy 
Priority order: 
1.​ Persistent loop — agent accepts instructions and runs continuously 
2.​ Basic tool calls — read_file and edit_file working end-to-end 
3.​ Surgical file editing — implement and verify your chosen strategy 
4.​ Context injection — external context accepted and used 
5.​ Multi-agent coordination — spawn two agents, merge outputs 
6.​ Ship rebuild — direct your agent at Ship, document everything 
7.​ Comparative analysis — write the full seven-section report 
 
Critical Guidance 
•​
Get surgical file editing working completely before moving to multi-agent 
•​
Test your edit strategy against files of varying size — behavior often breaks above 200 lines 
•​
Document every intervention during the Ship rebuild — this is your analysis data 
•​
Add tracing early — you need visibility to debug agent behavior 
•​
Do not mock responses at any stage — run against real code
