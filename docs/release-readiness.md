# SuperShip Release Readiness

## Current release posture

SuperShip is ready for internal demonstration and structured verification. The application now includes:

- shared document contracts
- API document CRUD
- frontend dashboard, explorer, editor, and parity console
- findings, traceability, and parity chat
- automated tests and workspace builds
- an end-to-end release verification script

## Demo flow

The recommended demo flow is:

1. open the dashboard and explain the unified document model
2. navigate to documents and show filtering and search
3. create or edit a document through the editor
4. open the parity console for the selected document
5. show proactive findings, trace timeline, and parity chat
6. run the release verification script to prove backend readiness

## Deployment shape

### API

- runtime: Node.js 20+
- start command: `pnpm --filter @supership/api start`
- default port: `3000`

### Web

- runtime: static assets generated by Vite
- build output: `apps/web/dist`
- expected API base URL: `http://localhost:3000` unless a global override is provided

## Verification commands

- `pnpm test`
- `pnpm build`
- `pnpm verify:release`

## Known limitations before production deployment

- no persistent database yet
- no authentication or authorization
- no multi-user collaboration runtime
- no durable event log or background processing layer
- no externalized environment configuration strategy yet

## Exit criteria for production readiness

Production readiness should require all of the following:

- PostgreSQL-backed persistence for documents and parity artifacts
- auth and visibility rules
- durable audit/event history
- richer graph associations across documents
- production deployment manifests and environment handling
- OpenAPI or equivalent published API contract
- e2e browser automation beyond API verification
